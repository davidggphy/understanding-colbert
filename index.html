Understanding ColBERT

ColBERT is a neural retrieval model that keeps one embedding per token and combines them with late interaction: queries and documents are encoded separately, and their token vectors only meet at scoring time. This page shows why that matters, with concrete examples.

1. The basic idea

What "late interaction" means: the query and the documents are encoded separately (like a dense retriever), but their token vectors are compared token-by-token at scoring time — the interaction happens late. Contrast this with a cross-encoder, which interacts early by jointly encoding the query and document together (accurate but expensive), and a dense retriever, whose vectors never interact at the token level at all.
BM25

Lexical matching

Scores overlap between query terms and document terms.

Excellent for names, IDs, rare terms and exact wording.

Dense retriever

One vector per passage

Compresses the whole query and passage into one vector each (also called a bi-encoder).

Good semantic matching, but fine details can get blurred.

ColBERT

One vector per token

Keeps token-level representations and matches them at query time.

More precise than a single-vector retriever, cheaper than a cross-encoder.

2. Example: paraphrases

Query:

cancel employment agreement

Relevant document:

procedure for terminating a work contract

BM25 view

There is almost no exact term overlap:

  • cancel ≠ terminating
  • employment ≠ work
  • agreement ≠ contract

BM25 may under-rank this passage unless analyzers, synonyms or expansion are added.

ColBERT view

Each query token can independently find its best semantic match:

  • cancel → terminating
  • employment → work
  • agreement → contract

This is the key benefit of token-level late interaction.

3. MaxSim, step by step

For every query token, ColBERT finds the most similar token in the document, then sums those maxima.

Here qᵢ and dⱼ are token embeddings, and the dot product qᵢ · dⱼ measures their similarity — the vectors are normalized, so this is cosine similarity: values near 1 mean "very similar in meaning", values near 0 mean "unrelated".

score(q,d) = Σᵢ maxⱼ ( qᵢ · dⱼ )
Query token ↓ / Document token → procedure terminating work contract Row max
cancel 0.18 0.91 0.12 0.34 0.91
employment 0.08 0.24 0.86 0.45 0.86
agreement 0.11 0.31 0.28 0.94 0.94
Final ColBERT score 2.71

The function words for and a are omitted from the matrix for clarity — they match nothing strongly and barely affect the score.

Read one row at a time: for cancel, the best document token is terminating (0.91), so cancel contributes 0.91 to the total — the other similarities in that row are simply ignored. Each query token contributes only its single best match, and the final score is the sum of those maxima: 0.91 + 0.86 + 0.94 = 2.71.

This is the "interaction" in late interaction: the query and document were encoded separately, but their token vectors meet here, in this matrix, at scoring time.

4. Why one dense vector can lose detail

Dense retriever: one vector per passage

A passage such as:

“The company may terminate an employee’s contract after a disciplinary review.”

is compressed into one vector. The overall topic may be captured, but distinct concepts can become entangled.

Concretely: a query like “how to appeal a disciplinary decision” and this passage both land near the same “HR / contract termination” region of vector space, so the single vector can't tell whether disciplinary review is the point of the passage or just mentioned in passing.

ColBERT

Separate contextual token vectors preserve local distinctions:

company terminate employee contract disciplinary review

Different query terms can match different parts of the passage.

ColBERT can be used as a first-stage retriever with a specialized index, or as a reranker over candidates returned by BM25 or dense search.

7. Practical trade-offs

Property BM25 Dense retriever ColBERT Cross-encoder
Exact matchingExcellentModerateGoodGood
Semantic matchingLimitedGoodVery goodExcellent
Fine-grained token matchingLexical onlyNoYesYes
Index sizeSmallMediumLargeNo reusable vector index
Query costLowLowMediumHigh

Why ColBERT's index is large: it stores one vector per token instead of one per passage — easily 10–100× more vectors. In practice this is mitigated with small per-token dimensions and aggressive compression: ColBERTv2's residual compression and the PLAID engine bring index size and query speed close to single-vector systems.

"No reusable vector index" for cross-encoders means they cannot pre-compute anything per document: the full model must re-run on every query–document pair at query time, which is why they are only practical for reranking a small candidate set.

8. The mental model

Think of BM25 as matching words, a dense retriever as matching the overall meaning, and ColBERT as matching the meaning of each important query token against the best part of the document.

What this page simplified: real ColBERT tokens are subword pieces rather than whole words; queries are padded with special [MASK] tokens that also participate in matching (a form of soft query expansion); and embeddings are normalized so dot products are cosine similarities.
